# Mnemosyne v7.2 Skill Evidence Flow Design

> Version: Draft | Date: 2026-05-10 | Status: Design Document  
> Builds on v7.1 bilateral Skill Evolution.

---

## 1. Thesis

Skill improvement must be evidence-backed, not automatic.

```text
Controlled trial -> evidence flow -> re-evaluation -> promote / revise / downgrade
```

v7.1 answered whether a skill can be called `evolved`. v7.2 should answer what happens after an evolved or approved skill is used in real work.

The correct model is not:

```text
default injection -> automatic learning -> automatic improvement
```

The correct model is:

```text
controlled use -> feedback write-back -> closed-loop calibration -> measured lifecycle decision
```

---

## 2. Terminology

Use these terms consistently:

| Term | Meaning |
|---|---|
| Evidence flow | Real usage outcomes flow back into the graph and test set. This is the main concept. |
| Feedback write-back | The engineering action that writes feedback nodes, edges, counters, and test prompts. |
| Closed-loop calibration | The broader process that uses evidence flow to adjust triggers, procedures, status, and future evaluations. |

Avoid casual terms such as `回血` in system documents. They are useful in conversation but too vague for governance.

---

## 3. Lifecycle Context

v7.1 lifecycle:

```text
embryo -> draft -> tested -> evolved -> approved -> deprecated
                 \-> needs_revision
                 \-> rejected
```

v7.2 adds the usage loop after `evolved` and `approved`:

```text
evolved
  -> controlled trial
  -> evidence flow
  -> re-evaluation
  -> approved / needs_revision / rejected

approved
  -> default injection
  -> sampling audit or failure-triggered audit
  -> evidence flow
  -> keep approved / needs_revision / deprecated
```

Key rule:

```text
evolved means measured potential.
approved means default injection permission.
```

---

## 4. Goals

### In Scope

- Define controlled trial for low-risk evolved skills.
- Define feedback write-back semantics.
- Define how successful, failed, partial, and misleading usage updates the graph.
- Define how usage evidence becomes future test prompts.
- Define sampling audits for approved skills.
- Define downgrade paths from `approved` back to `needs_revision` or `deprecated`.

### Out of Scope

- Generic autonomous agent runtime.
- Browser, desktop, mobile, IM, or arbitrary script automation.
- Automatic approval without policy or human gate.
- Continuous online learning that mutates skills without re-evaluation.
- UI/dashboard implementation.

---

## 5. Usage Modes

### Default Injection

Default injection is only for approved skills:

```text
status='approved' AND inject_enabled=true AND has verified evidence
```

Default injection should be conservative. It should retrieve only skills whose trigger patterns match the current task with high confidence.

### Trial Injection

Trial injection is for low-risk evolved skills:

```text
status='evolved' AND risk_level='low' AND mode='trial'
```

Trial injection must produce feedback. A trial without feedback is not evidence.

### Experimental Injection

Experimental injection is for research and debugging:

```text
mode='experimental'
```

Experimental use must be clearly marked and must not affect default injection eligibility unless it is converted into reviewed evidence.

---

## 6. Feedback Outcomes

Every feedback event should classify the usage outcome.

| Outcome | Meaning | Graph effect |
|---|---|---|
| success | The skill helped complete the task correctly. | Add `verified_by`, increment success counters. |
| partial | The skill was directionally useful but incomplete. | Add feedback node, consider `needs_revision`. |
| miss | The skill did not help. | Add `fails_on`, increment failure counters. |
| misleading | The skill caused wrong reasoning or bad action. | Add `fails_on`, create regression prompt, consider downgrade. |
| trigger_mismatch | The skill was injected for the wrong task. | Reduce trigger precision, consider trigger revision. |

Do not collapse all failures into one bucket. A useless skill, an incomplete skill, and a wrongly triggered skill need different repairs.

---

## 7. Feedback Write-Back

The write-back operation should update structured evidence, not just append notes.

Minimum write-back fields:

```text
skill_id
task_context
used_as: approved | trial | experimental
outcome: success | partial | miss | misleading | trigger_mismatch
verification_result
note
created_at
```

Graph updates:

```text
success -> verified_by edge
miss -> fails_on edge
partial -> feedback node + possible needs_revision edge
misleading -> fails_on edge + regression case
trigger_mismatch -> trigger revision signal
```

Counters:

```text
trial_count
trial_success_count
trial_failure_count
last_trial_at
```

## 8. Evidence To Test Prompts

Important failures should become future evaluation prompts.

Convert feedback into `test-prompts.json` when:

- The skill caused a wrong answer.
- The skill missed a case it claims to cover.
- The trigger matched too broadly.
- The user corrected the skill's behavior.
- The same failure pattern appears more than once.

Prompt record shape:

```json
{
  "id": "usage-failure-001",
  "prompt": "Original or minimized task context",
  "expected": "What the skill should have caused the agent to do",
  "tags": ["usage_feedback", "regression"],
  "status": "active"
}
```

The prompt should be minimized. Do not store private, irrelevant, or high-risk content when a smaller reproduction is enough.

---

## 9. Approved Skill Audits

Approved does not mean permanent.

Approved skills should be audited by two mechanisms:

### Sampling Audit

Occasionally evaluate an approved skill even when no failure is obvious.

Sampling rate should depend on risk:

| Risk | Suggested audit policy |
|---|---|
| low | Rare sampling. |
| medium | Regular sampling or after important tasks. |
| high | No default injection without explicit policy; frequent review if allowed. |

### Failure-Triggered Audit

Immediately audit when:

- The user corrects the result.
- The task fails after the skill was injected.
- The skill contradicts newer evidence.
- The skill triggers in an unrelated context.
- A regression prompt fails.

---

## 10. Lifecycle Decisions From Usage Evidence

Usage feedback should drive measured decisions.

```text
success trend + clean audits -> keep approved
trial success + bilateral re-eval pass -> eligible for approval
partial trend -> needs_revision
miss trend -> needs_revision or rejected
misleading or unsafe behavior -> deprecated or rejected
trigger mismatch trend -> trigger revision before more injection
```

Hard rule:

```text
Do not mutate a skill into a better-looking version without rerunning evaluation.
```

Evidence can propose a mutation. Evaluation decides whether the mutation stays.

---

## 11. Minimal v7.2 Acceptance Criteria

v7.2 is done when these are true:

1. `memory_skill_feedback` records structured outcomes, not just free-text ratings.
2. Feedback writes correct graph edges for success, miss, partial, misleading, and trigger mismatch.
3. Failure feedback can create or update `test-prompts.json` entries.
4. Approved skills can be sampled for audit.
5. Failure-triggered audit can downgrade an approved skill to `needs_revision` or `deprecated`.
6. Trial injection requires feedback before it can contribute to approval evidence.
7. Tests cover success, partial, miss, misleading, trigger mismatch, prompt creation, and downgrade.

---

## 12. Non-Negotiables

- Default injection is not automatic learning.
- Trial usage without feedback is not evidence.
- Evidence flow must preserve failure signals.
- User correction is high-weight evidence.
- Approved skills must remain auditable.
- Skills can be downgraded after approval.
- Private data must be minimized before becoming a test prompt.
- A skill improves only after evidence-backed mutation and re-evaluation.

---

## 13. One-Line Summary

```text
Mnemosyne skills do not get smarter by being injected; they get smarter when controlled usage produces evidence, that evidence flows back, and re-evaluation proves the revision works.
```
