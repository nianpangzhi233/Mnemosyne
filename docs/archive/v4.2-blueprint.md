> 基于 OpenViking、Claude-Mem、opencode-autosave-conversation、Mem0、Cloudflare Agent Memory、Hindsight 
> 等项目的深度调研，结合 Grill Me 结构化访谈达成的共识。

## 一、设计理念转变

### v4.1 → v4.2 的核心认知升级

**v4.1 的问题：** 写入的经验本身就是高度浓缩的（"gzip解压必须先解压"），这已经是 L0 精华了。
强行加 L0/L1/L2 分层等于三层都一样，没有意义。

**v4.2 的正确分层：**
- **L2（原始材料）：** 完整的对话片段——用户输入、AI 操作序列、工具调用、代码变更、错误日志
- **L1（概要）：** ~500 token 的事件概要（发生了什么、怎么解决的）
- **L0（精华）：** ~100 token 的可复用原理（当前的经验格式）

**L2 是原材料，做梦时蒸馏成 L0。** L0 是结果，不是起点。

### 记忆全周期

目标不只是"错题库"，而是 AI 的全周期记忆：
- 记住做对了什么（成功经验）
- 记住做错了什么（失败教训）
- 记住为什么选 A 不选 B（决策过程）
- 记住用户的偏好和风格
- 记住项目特定的模式和约定

## 二、记忆写入：双通道架构

### 通道 1：MCP Server（Agent 主动写入）

**定位：** 通用跨平台接口，任何支持 MCP 的 Agent 都能用

```
Agent (任意平台: Claude Code / OpenCode / Cursor / VS Code)
    │
    ├── 完成重要任务 → memory_write("经验内容", type="experience")
    ├── 被纠正 → memory_write("纠正内容", type="correction", contradicts="旧节点ID")
    ├── 会话开始 → memory_inject(context="工作目录")
    ├── 遇到问题 → memory_search("关键词", layer="L0")
    ├── 需要详情 → memory_detail(["id1", "id2"])
    │
    ▼
Mnemosyne MCP Server (stdio / SSE)
    ├── memory_write    → 写入 L2 原始材料 + 自动生成 L0/L1
    ├── memory_search   → 按 layer 返回（L0/L1/L2）
    ├── memory_inject   → 注入联想链（L0 摘要）
    └── memory_detail   → 按 ID 批量拉取完整内容
```

**Agent 怎么知道该调？** system prompt / AGENTS.md / CLAUDE.md 里写规则。
跟现在让 AI 调 `graph_write` 一样的逻辑，只是从 CLI 换成 MCP 工具调用。

### 通道 2：对话日志后处理（兜底自动写入）

**定位：** Agent 不调 MCP 也不丢记忆。定期扫描 opencode 对话日志，提取有价值片段作为 L2 原材料。

**触发时机：** 一天 2 次（跟做梦绑定）
- **12:00** — 扫描上午对话 + 做梦，下午能用上新经验
- **03:00** — 扫描全天对话 + 做梦，第二天醒来经验已整理

**数据源：** `~/.local/share/opencode/storage/`（opencode 对话日志）

**处理流水线：**

```
opencode/storage/
    │
    ▼
Step 1: 增量扫描
    ├── 记录上次扫描位置，只处理新增的 session
    └── 按 session 目录分组
    │
    ▼
Step 2: 片段切分
    ├── 按 AI Stop 事件切分（每次 AI 停下来等用户 = 一个片段边界）
    └── 兜底：连续 30 分钟无交互也视为边界
    │
    ▼
Step 3: 片段级过滤（规则层，零成本）
    ├── 丢弃：消息数 < 3 轮的片段
    ├── 丢弃：纯问答（无工具调用）
    ├── 保留：任务完成序列（用户需求 → AI 操作 → 结果）
    ├── 保留：决策过程（"用方案A不用方案B"）
    ├── 保留：错误 + 修复（失败尝试 + 最终解决）
    ├── 保留：用户反馈（"不对"/"重做"/"这个好"）
    └── 保留：代码变更摘要（改了哪些文件、为什么）
    │
    ▼
Step 4: 内容级去噪（规则层，零成本）
    ├── 工具调用输出精简：git status 只保留命令+结果摘要，去掉文件列表
    ├── 重复内容去重：AI 套话（"我来帮你"/"让我看看"）只保留一次
    ├── 系统噪音去除：warning、deprecation notice 直接丢弃
    └── 中间状态合并：连续 read 同一文件不同部分 → "查看了文件 X"
    │
    ▼
Step 5: 写入 L2 原始材料
    └── graph_write(content=去噪后片段, type="raw", source="log_scan")
    │
    ▼
Step 6: LLM 蒸馏（在做梦时执行）
    ├── 判断"这条 L2 是否值得长期存储"
    ├── 值得 → 蒸馏为 L0 精华 + L1 概要
    └── 不值得 → 标记为 cold，下次清洗删除
```

**新增文件：**
```
scripts/log_scanner/
├── __init__.py
├── scanner.py          -- 增量扫描 + 片段切分
├── filter.py           -- 片段级过滤 + 内容级去噪
└── run_scan.py         -- CLI 入口（手动触发 / 定时调用）
```

## 三、L0/L1/L2 分层上下文

### Schema 变更

```sql
ALTER TABLE nodes ADD COLUMN abstract TEXT;      -- L0: ~100 token
ALTER TABLE nodes ADD COLUMN overview TEXT;      -- L1: ~500 token
-- content 保持不变                               -- L2: 完整原文
```

### 各层生成策略

| 层级 | 内容 | 生成时机 | 生成方式 |
|------|------|---------|---------|
| L2 | 完整原始材料 | 写入时 | 直接存储 MCP 传入内容 / 日志扫描片段 |
| L1 | ~500 token 概要 | 写入时（规则）/ 做梦时（LLM 优化） | 规则：content[:500] + principle + tags |
| L0 | ~100 token 精华 | 写入时（规则）/ 做梦时（LLM 优化） | 规则：content[:100] + principle（如有） |

**LLM 只在做梦时介入优化 L0/L1 质量**（与 v4.1 设计理念一致）。

### 搜索分级返回

```
--layer L0   → {id, abstract, similarity, score, tier}
--layer L1   → {id, abstract, overview, similarity, score, tier, principle}
--layer L2   → 完整内容（默认，向后兼容）

inject 模式   → 用 L0 代替当前 80 字符硬截断
```

## 五、做梦增强

### 做梦频率调整

v4.1: 1 次/天（03:00）
v4.2: **2 次/天**（12:00 + 03:00）

### 新增 Phase：LogScanPhase

插入到现有 Phase 0（Snapshot）之后：

| 旧编号 | 新编号 | Phase | 说明 |
|--------|--------|-------|------|
| — | 0.5 | **LogScan** | 扫描对话日志 → 过滤去噪 → 写入 L2 原始材料 |
| 1 | 1 | Snapshot | 不变 |
| ... | ... | ... | ... |
| 8 | 9 | LLM Review | 不变，新增：对 L2 原始材料蒸馏为 L0/L1 |

### 新增 Phase：DistillPhase

在做梦的 LLM Review 阶段，新增 L2 → L0/L1 蒸馏：
- 扫描所有 type="raw" 的 L2 节点
- 用 LLM 判断是否值得长期存储
- 值得 → 生成 L0 精华 + L1 概要，type 改为 "experience"
- 不值得 → 标记 cold

## 八、与 v4.1 的兼容性

- 所有新功能向后兼容
- `--layer` 不传时默认 L2（当前行为）
- MCP Server 为独立进程，不影响 CLI 使用
- 对话日志扫描为可选模块，没有 opencode 日志也不报错
- 数据库迁移为 ALTER TABLE（非重建），数据零丢失
- 做梦频率从 1 次 → 2 次，但 Phase 编号不变（LogScan 和 Distill 为新增可选 Phase）

## 八、风险点

| 风险 | 应对 |
|------|------|
| opencode 日志格式变动 | scanner 做容错解析，字段缺失跳过不崩 |
| L2 原始材料膨胀 | 冷热分层 + 做梦蒸馏 + audit 清洗 |
| Agent 不调 MCP 工具 | 对话日志扫描兜底，双通道互不依赖 |
| MCP Server 加载嵌入模型慢 | 首次调用时加载，后续复用单例 |
| 内容去噪误删有用信息 | 规则层只删明确的噪音，LLM 层做精细判断 |

## 十、决策记录

以下为 Grill Me 访谈达成的共识：

| # | 决策 | 结论 |
|---|------|------|
| 1 | L2 原始材料来源 | MCP 主动写入 + 对话日志后处理兜底，两者共存 |
| 2 | Hook 实现策略 | MCP Server 为主（通用跨平台），Hook 为可选增强 |
| 3 | "Agent 不调工具"的解法 | 对话日志后处理兜底，定期扫描 opencode storage |
| 4 | 日志扫描频率 | 一天 2 次（12:00 + 03:00），跟做梦绑定 |
| 5 | 片段切分方式 | 按 AI Stop 事件切分，30 分钟无交互兜底 |
| 6 | 过滤策略 | 两层：规则层（片段级 + 内容级去噪）+ LLM 层（做梦时精炼） |
| 7 | 记忆目标 | 全周期 AI 记忆（不只是错题库），成功/失败/决策/偏好全记 |
