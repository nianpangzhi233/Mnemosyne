# Mnemosyne v7.1 Bilateral Skill Evolution Design

> Version: Draft | Date: 2026-05-09 | Status: Design Document  
> Supersedes the weak interpretation of v7.0 `Darwin evolution` as static dry-run scoring.

---

## 1. Thesis

Mnemosyne Skill Evolution is bilateral.

```text
Darwin proves the skill improves agent behavior.
Mnemosyne proves the skill is evidence-backed, safe, and suitable for injection.
Both must pass before a skill can be called evolved.
```

The v7.0 implementation created the governance foundation, but its Darwin section is not a real evolution loop yet. Static field completeness and Markdown quality are not enough.

Correct definition:

```text
evolved = Darwin live-test improvement + Mnemosyne graph-governance pass
```

Wrong definition:

```text
evolved = SKILL.md exists + static score is high
```

---

## 2. Goals

### In Scope

- Define the bilateral scoring model.
- Define the real Darwin execution loop.
- Define the Mnemosyne graph-governance scorer.
- Define how both sides combine into lifecycle decisions.
- Define required database records and file artifacts.
- Define the minimum end-to-end acceptance case.

### Out of Scope

- Generic autonomous agent runtime.
- Browser, desktop, mobile, IM, or arbitrary script automation.
- Fully automatic production approval without human or strict policy gate.
- UI/dashboard implementation.

---

## 3. Current Problem

The weak v7.0 interpretation effectively did this:

```text
experience cluster -> embryo -> draft -> SKILL.md mirror -> dry-run score -> evolved
```

This is insufficient because:

- This is a historical failure mode, not the current target pipeline.
- The agent never runs the task with and without the skill.
- There is no baseline output.
- There is no with-skill output.
- There is no independent evaluator comparing the two.
- There is no mutation loop that edits the skill and reruns tests.
- There is no keep/revert ratchet.
- Mnemosyne graph evidence is not part of the evolved decision strongly enough.

Therefore, v7.0 should be described as:

```text
Skill Memory governance foundation, not complete Skill Evolution.
```

---

## 4. Correct Pipeline

```text
experience cluster
  -> embryo
  -> draft
  -> Darwin live execution loop
  -> Mnemosyne graph-governance scoring
  -> bilateral decision
  -> evolved candidate
  -> approval gate
  -> approved injection
  -> feedback graph update
```

### Stage Responsibilities

| Stage | Owner | Question Answered | Output |
|---|---|---|---|
| Graph Embryo | Mnemosyne graph | Is there a repeatable pattern? | `embryo` |
| LLM Development | Draft writer | Can this pattern become an operational skill? | `draft` |
| Darwin Execution | Agent test loop | Does the skill improve real task behavior? | `darwin_score`, `live_test_delta` |
| Mnemosyne Scoring | Graph governance | Is the skill evidence-backed and safe to inject? | `mnemosyne_score`, risk decision |
| Bilateral Decision | Policy combiner | Do both sides pass? | `tested`, `evolved`, `needs_revision`, or `rejected` |
| Approval | Human or strict gate | May this enter default injection? | `approved` |

---

## 5. Lifecycle

v7.1 should use a more precise lifecycle than v7.0.

```text
embryo -> draft -> tested -> evolved -> approved -> deprecated
                 \-> needs_revision
                 \-> rejected
```

### embryo

Graph-discovered candidate from recurring memories or experience clusters.

Requirements:

- Has source memory nodes.
- Has a cluster reason.
- Has candidate trigger patterns.
- Does not inject.

### draft

LLM-developed operational skill draft.

Requirements:

- Has trigger patterns.
- Has preconditions.
- Has procedure.
- Has verification method.
- Has failure modes.
- Has risk level.
- Has source evidence node IDs.
- Does not inject by default.

### tested

The skill has entered Darwin live testing.

Requirements:

- Has `test-prompts.json` or DB-backed test prompts.
- Has at least one baseline run.
- Has at least one with-skill run.
- Has judge output.

`tested` does not mean good. It only means measured.

### evolved

The skill passed both sides.

Requirements:

- Darwin side passed live execution tests.
- Mnemosyne side passed graph-governance checks.
- `live_test_delta > 0`.
- No critical regression.
- Has `SKILL.md` mirror.
- Has reproducible evolution records.
- Not injected by default unless approved.

### approved

The skill is allowed into default injection.

Requirements:

- `status='approved'`.
- `inject_enabled=1`.
- `status` was previously `evolved` or has a documented manual override.
- Has verified evidence.
- Has clean file hash.
- Has acceptable risk level.

### needs_revision

The skill has potential but failed one side.

Examples:

- Darwin live test did not improve enough.
- Mnemosyne evidence score is weak.
- Trigger patterns are too broad.
- Feedback produced `needs_revision` edges.

### rejected

The skill should not continue.

Examples:

- Live tests are worse than baseline.
- Evidence is incoherent.
- Risk is high and unjustified.
- The skill contains unsafe or private content.

### deprecated

Soft removal from use.

Rules:

- Never inject.
- Keep node.
- Keep artifact.
- Keep evidence.
- Keep file mirror, marked deprecated.
- Use purge only for secrets, privacy, or illegal-sensitive content.

## 6. Darwin Side: Live Execution Evolution

Darwin is not a static scorer. It is an agent execution optimizer.

### 6.1 Inputs

- `SKILL.md`
- `test-prompts.json`
- Target agent runner
- Judge runner
- Current best score
- Current best file hash

### 6.2 Test Prompts

Each skill needs 2-3 prompts covering common usage.

Example:

```json
[
  {
    "id": "gzip-json-1",
    "prompt": "A Node.js API proxy receives a gzip encoded JSON body and JSON.parse fails. What should I check first?",
    "expected": "Check Content-Encoding and gunzip before parsing JSON."
  },
  {
    "id": "gzip-json-2",
    "prompt": "An OpenAI-compatible proxy receives garbled request bytes before JSON.parse. Diagnose the likely root cause.",
    "expected": "Inspect request headers, especially Content-Encoding, before assuming plain JSON."
  }
]
```

Test prompts must be grounded in source memories, failures, triggers, procedures, verification criteria, or known failure modes. Structural validation should filter only clear garbage or unsafe content; validated prompts enter the executable feedback loop directly. Bad prompts are rejected by baseline-vs-skill evaluation, judge scoring, regression checks, and Mnemosyne governance.

### 6.3 Baseline Run

Run the target agent without injecting the skill.

Output:

- `baseline_output`
- `baseline_actions`
- `baseline_score`
- `baseline_failure_modes`

Purpose:

```text
Measure what the agent would do without the skill.
```

### 6.4 With-Skill Run

Run the same prompt with the skill injected.

Output:

- `with_skill_output`
- `with_skill_actions`
- `with_skill_score`
- `skill_helped`
- `skill_harmed`

Purpose:

```text
Measure whether the skill changes behavior in the right direction.
```

### 6.5 Independent Judge

The judge should not be the same context that edited the skill.

Judge compares:

- Did the answer solve the user's intent?
- Is with-skill better than baseline?
- Did the skill introduce verbosity, wrong assumptions, or unsafe behavior?
- Did it over-trigger or force irrelevant procedures?
- Is the improvement specific to this skill?

Judge output:

```json
{
  "winner": "with_skill",
  "baseline_score": 6.5,
  "with_skill_score": 8.5,
  "delta": 2.0,
  "regression": false,
  "reason": "With skill checks Content-Encoding before JSON.parse; baseline guesses schema issues first."
}
```

### 6.6 Mutation Loop

If the skill is weak but promising, generate one targeted edit.

Rules:

- Modify one dimension per round.
- Prefer minimal edits.
- Do not change the skill's core purpose.
- Do not exceed 150% of previous file size without explicit approval.
- Record the reason for the mutation.

Allowed mutation targets:

- Trigger precision.
- Preconditions.
- Procedure clarity.
- Verification steps.
- Failure modes.
- Safety boundaries.

### 6.7 Ratchet

Only measurable improvements survive.

```text
if new_total_score > previous_best_score and no critical_regression:
    keep
else:
    revert
```

Revert should preserve audit history.

Preferred behavior:

- Git-backed file mirror: use `git revert` when committed.
- DB-only artifact: restore previous content and file hash, record a reverted run.

No destructive reset.

### 6.8 Darwin Score

Darwin score should combine structure and live behavior.

```text
darwin_score = structure_score * 0.35 + live_test_score * 0.50 + ratchet_score * 0.15
```

Suggested components:

| Component | Weight | Meaning |
|---|---:|---|
| `structure_score` | 35 | Is the skill clear and executable? |
| `live_test_score` | 50 | Does it improve task output in real tests? |
| `ratchet_score` | 15 | Did the latest mutation improve over the previous best? |

Hard caps:

- No test prompts: `darwin_score <= 40`.
- No baseline run: `darwin_score <= 50`.
- No with-skill run: `darwin_score <= 50`.
- No independent judge: `darwin_score <= 65`.
- Dry-run only: `darwin_score <= 70`.
- No positive delta: cannot pass Darwin.

## 7. Mnemosyne Side: Graph Governance Scoring

Mnemosyne scoring decides whether a skill is evidence-backed and safe for retrieval/injection.

### 7.1 Inputs

- Source memory nodes.
- `crystallized_from` edges.
- `verified_by` edges.
- `fails_on` edges.
- `needs_revision` edges.
- Similarity cluster density.
- Correction history.
- Feedback history.
- Trigger patterns.
- Risk level.
- Project/task tags.

### 7.2 Score Components

```text
mnemosyne_score = evidence * 0.25
                + cluster_quality * 0.20
                + verification * 0.20
                + trigger_precision * 0.15
                + feedback * 0.10
                + safety * 0.10
```

Suggested meanings:

| Component | Weight | Meaning |
|---|---:|---|
| `evidence` | 25 | Enough source memories and causal context? |
| `cluster_quality` | 20 | Are source nodes coherent rather than accidental neighbors? |
| `verification` | 20 | Has the skill been verified by outcomes or user confirmation? |
| `trigger_precision` | 15 | Will it fire only in relevant contexts? |
| `feedback` | 10 | Do trial outcomes support continued use? |
| `safety` | 10 | Is risk controlled and free of sensitive material? |

### 7.3 Hard Gates

Mnemosyne side fails immediately if:

- Source evidence is missing.
- Evidence nodes are unrelated or contradictory without resolution.
- Risk level is `high` and no human override exists.
- Trigger patterns are broad enough to pollute common contexts.
- Any source or file contains secrets/private data.
- There are strong unresolved `fails_on` edges.

### 7.4 Mnemosyne Score Caps

- No `crystallized_from`: `mnemosyne_score <= 40`.
- No source memory details: `mnemosyne_score <= 45`.
- No verification or feedback: `mnemosyne_score <= 70`.
- Any unresolved `needs_revision`: `mnemosyne_score <= 75`.
- High risk without human approval: cannot pass.

## 8. Bilateral Decision

### 8.1 Required Fields

Every evolution decision must record:

- `darwin_score`
- `mnemosyne_score`
- `live_test_delta`
- `regression_count`
- `eval_mode`
- `judge_reason`
- `decision`
- `decision_reason`

### 8.2 Decision Policy

```text
if darwin_score >= 80
and mnemosyne_score >= 80
and live_test_delta > 0
and regression_count == 0
and no_hard_gate_failed:
    decision = evolved
elif skill_has_potential:
    decision = needs_revision
else:
    decision = rejected
```

### 8.3 Status Rules

- `draft` can become `tested` after baseline and with-skill runs exist.
- `tested` can become `evolved` only after bilateral pass.
- `evolved` can become `approved` only after approval gate.
- `approved` can become `deprecated` after failures, user rejection, or safety issues.
- Dry-run cannot produce `evolved`; dry-run can only produce `needs_revision` or `format_checked` metadata.

### 8.4 Approval Gate

Approval is not evolution. Approval is production injection permission.

Requirements:

- Status is `evolved`, or manual override is explicit.
- `SKILL.md` exists.
- File hash matches DB.
- Has at least one `verified_by` edge.
- Has positive Darwin live-test result.
- Has passing Mnemosyne score.
- Risk is low/medium, or human override exists.

## 9. Data Model Additions

v7.0 already has `skill_artifacts` and `skill_evolution_runs`. v7.1 should expand the experiment records.

### 9.1 skill_test_prompts

```sql
CREATE TABLE IF NOT EXISTS skill_test_prompts (
    id TEXT PRIMARY KEY,
    skill_id TEXT NOT NULL,
    prompt_id TEXT NOT NULL,
    prompt TEXT NOT NULL,
    expected TEXT,
    tags TEXT DEFAULT '[]',
    metadata TEXT DEFAULT '{}',
    status TEXT DEFAULT 'active',
    approved_by TEXT,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(skill_id, prompt_id)
);
```

### 9.2 skill_eval_runs

```sql
CREATE TABLE IF NOT EXISTS skill_eval_runs (
    id TEXT PRIMARY KEY,
    skill_id TEXT NOT NULL,
    prompt_id TEXT,
    round INTEGER DEFAULT 0,
    eval_mode TEXT NOT NULL,
    baseline_output TEXT,
    with_skill_output TEXT,
    judge_output TEXT,
    baseline_score REAL,
    with_skill_score REAL,
    live_test_delta REAL,
    regression INTEGER DEFAULT 0,
    darwin_score REAL,
    mnemosyne_score REAL,
    decision TEXT,
    decision_reason TEXT,
    file_hash_before TEXT,
    file_hash_after TEXT,
    kept INTEGER DEFAULT 0,
    reverted INTEGER DEFAULT 0,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP
);
```

### 9.3 skill_mutations

```sql
CREATE TABLE IF NOT EXISTS skill_mutations (
    id TEXT PRIMARY KEY,
    skill_id TEXT NOT NULL,
    eval_run_id TEXT,
    round INTEGER DEFAULT 0,
    target_dimension TEXT,
    reason TEXT,
    patch_summary TEXT,
    file_hash_before TEXT,
    file_hash_after TEXT,
    status TEXT,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP
);
```

### 9.4 skill_artifacts New Fields

Recommended additions:

```sql
ALTER TABLE skill_artifacts ADD COLUMN latest_darwin_score REAL;
ALTER TABLE skill_artifacts ADD COLUMN latest_mnemosyne_score REAL;
ALTER TABLE skill_artifacts ADD COLUMN latest_live_test_delta REAL;
ALTER TABLE skill_artifacts ADD COLUMN latest_eval_mode TEXT;
ALTER TABLE skill_artifacts ADD COLUMN latest_decision TEXT;
ALTER TABLE skill_artifacts ADD COLUMN latest_decision_reason TEXT;
```

## 10. File Artifacts

Each skill directory should contain:

```text
skills/<slug>/
  SKILL.md
  test-prompts.json
  results.tsv
```

### SKILL.md

Human-readable skill package.

### test-prompts.json

Benchmark prompts for Darwin live testing.

### results.tsv

Append-only human-readable experiment log.

Columns:

```text
timestamp	round	skill	old_score	new_score	darwin_score	mnemosyne_score	delta	decision	eval_mode	note
```

DB remains source of truth. Files are mirrors and review artifacts.

## 11. Runner Interfaces

v7.1 should define interfaces before implementation.

### 11.1 AgentRunner

```python
class AgentRunner:
    def run(self, prompt: str, skill_content: str | None = None) -> AgentRunResult:
        ...
```

Responsibilities:

- Run baseline without skill.
- Run with-skill using injected skill content.
- Return output text and optional action trace.

### 11.2 JudgeRunner

```python
class JudgeRunner:
    def judge(self, prompt: str, expected: str, baseline: str, with_skill: str) -> JudgeResult:
        ...
```

Responsibilities:

- Compare baseline and with-skill.
- Produce scores and reason.
- Flag regressions.

### 11.3 SkillMutator

```python
class SkillMutator:
    def mutate(self, skill_content: str, eval_result: JudgeResult, target_dimension: str) -> MutationResult:
        ...
```

Responsibilities:

- Modify one dimension.
- Preserve core purpose.
- Produce patch summary.

### 11.4 MnemosyneSkillScorer

```python
class MnemosyneSkillScorer:
    def score(self, skill_id: str) -> MnemosyneScoreResult:
        ...
```

Responsibilities:

- Score evidence, cluster quality, verification, trigger precision, feedback, and safety.
- Apply hard gates.

### 11.5 EvolutionDecisionEngine

```python
class EvolutionDecisionEngine:
    def decide(self, darwin: DarwinScoreResult, mnemosyne: MnemosyneScoreResult) -> EvolutionDecision:
        ...
```

Responsibilities:

- Merge both sides.
- Decide status.
- Explain the decision.

## 12. Minimum E2E Acceptance Case

Use the gzip JSON body skill because it came from a real repeated failure.

### Source Memories

- API gateway receives gzip-compressed request body.
- Direct `JSON.parse` fails or sees garbled bytes.
- Correct fix is to inspect `Content-Encoding` and gunzip before parsing.

### Test Prompt

```text
A Node.js API proxy receives request bytes that look garbled, and JSON.parse fails. What should I check before assuming the JSON schema is wrong?
```

### Expected Baseline Weakness

Without skill, agent may guess:

- malformed JSON
- bad schema
- charset issue
- body parser config

### Expected With-Skill Improvement

With skill, agent should say:

- inspect request headers first
- check `Content-Encoding: gzip`
- gunzip before `JSON.parse`
- do not assume plain JSON

### Acceptance

The skill may become `evolved` only if:

- Baseline and with-skill outputs are both recorded.
- Judge marks with-skill as better.
- `live_test_delta > 0`.
- Mnemosyne score passes evidence and safety gates.
- Evolution run is recorded.
- File hash is synced.

## 13. Implementation Milestones

### M1: Design and Schema

- Add this design document.
- Add migrations for `skill_test_prompts`, `skill_eval_runs`, and `skill_mutations`.
- Add fields to `skill_artifacts`.

Acceptance:

- Migration is idempotent.
- Existing v7.0 DB upgrades safely.

### M2: MnemosyneSkillScorer

- Implement graph-governance score.
- Apply hard gates.
- Add unit tests for scoring.

Acceptance:

- Skills without evidence cannot pass.
- High-risk skills cannot pass without override.
- Strong verified evidence increases score.

### M3: Darwin Test Prompt Support

- Generate or store test prompts.
- Mirror `test-prompts.json`.
- Require test prompts before live evaluation.

Acceptance:

- No test prompts means no Darwin pass.

### M4: Baseline and With-Skill Runner

- Define runner interface.
- Implement local LLM runner or MCP-backed runner.
- Record outputs.

Acceptance:

- Same prompt can be run baseline and with-skill.
- Outputs are persisted.

### M5: Independent Judge

- Implement judge interface.
- Compare baseline and with-skill.
- Record scores and reasons.

Acceptance:

- Judge can select winner and explain why.
- No independent judge caps Darwin score.

### M6: Mutation and Ratchet

- Generate one targeted skill edit.
- Rerun tests.
- Keep only improvements.
- Revert non-improvements.

Acceptance:

- A worse mutation is reverted.
- A better mutation is kept and recorded.

### M7: Bilateral Decision

- Combine Darwin and Mnemosyne scores.
- Add `tested`, `needs_revision`, and `rejected` handling.
- Block `evolved` without bilateral pass.

Acceptance:

- Dry-run cannot create `evolved`.
- Darwin pass without Mnemosyne pass cannot create `evolved`.
- Mnemosyne pass without Darwin pass cannot create `evolved`.

### M8: Approval Gate Hardening

- Approval requires bilateral pass.
- Approval checks file hash.
- Approval checks verified evidence.

Acceptance:

- Draft cannot be approved accidentally.
- Evolved with stale file hash cannot be approved.

---

## 14. Non-Negotiable Rules

1. Static scoring cannot produce `evolved`.
2. Dry-run is a fallback label, not a promotion path.
3. No baseline means no Darwin pass.
4. No with-skill run means no Darwin pass.
5. No independent judge means Darwin score is capped.
6. No source evidence means no Mnemosyne pass.
7. High-risk skills require human approval before injection.
8. Score regressions must be reverted or explicitly accepted by a human.
9. `approved` means injection permission, not evolution quality.
10. `evolved` means measured improvement plus graph trust.

---

## 15. Summary

v7.0 built the skeleton:

```text
skill artifacts, lifecycle, file mirrors, MCP tools, injection gates
```

v7.1 must build the heart:

```text
Darwin live execution improvement + Mnemosyne graph trust scoring
```

Final rule:

```text
Darwin says it works.
Mnemosyne says it is trustworthy.
Only then may the system say it evolved.
```
