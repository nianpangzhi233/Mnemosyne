# Mnemosyne v7.2 Skill Evidence Flow Development Plan

> Version: Draft | Date: 2026-05-10 | Status: Ready for Implementation  
> Source design: `docs/v7.2-skill-evidence-flow.md`

---

## 1. Development Goal

Implement the v7.2 evidence-flow loop without building a generic agent runtime.

Target flow:

```text
controlled trial -> evidence flow -> re-evaluation -> promote / revise / downgrade
approved injection -> sampling audit or failure-triggered audit -> evidence flow -> keep / revise / deprecate
```

The system must treat skill usage as governed evidence, not automatic learning.

---

## 2. Current Baseline

Already exists:

- `SQLiteStore.skill_feedback(...)`
- `memory_skill_feedback` MCP tool.
- `verified_by`, `fails_on`, and `needs_revision` graph edges.
- Trial counters on `skill_artifacts`.
- `skill_test_prompts` table and `test-prompts.json` mirroring.
- `SkillEvolutionRunner` for baseline vs with-skill evaluation.
- `decide_skill_evolution(...)` for bilateral lifecycle decision.

Current gaps:

- Feedback uses broad ratings: `helpful`, `not_helpful`, `misleading`, `partially_useful`.
- There is no explicit `trigger_mismatch` outcome.
- Feedback does not create regression test prompts.
- Trial feedback is not enforced as a requirement before approval evidence.
- Approved skills are not sampled for audit.
- Failure-triggered audit is not formalized.
- Downgrade from `approved` based on usage evidence is incomplete.

---

## 3. Non-Goals

Do not implement these in v7.2:

- Browser, desktop, mobile, IM, or arbitrary script automation.
- Generic autonomous execution agent.
- UI/dashboard.
- Automatic skill mutation without rerunning evaluation.
- Automatic approval without policy or human gate.
- Any external API dependency in core tests.

---

## 4. Data Model Changes

### 4.1 Add Skill Usage Feedback Table

Add `skill_usage_feedback` as the structured record. Keep `nodes(type='skill_feedback')` for graph compatibility.

```sql
CREATE TABLE IF NOT EXISTS skill_usage_feedback (
    id TEXT PRIMARY KEY,
    skill_id TEXT NOT NULL,
    feedback_node_id TEXT,
    task_context TEXT,
    used_as TEXT NOT NULL,
    outcome TEXT NOT NULL,
    rating TEXT,
    verification_result TEXT,
    note TEXT,
    created_prompt_id TEXT,
    audit_required INTEGER DEFAULT 0,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY(skill_id) REFERENCES nodes(id),
    FOREIGN KEY(feedback_node_id) REFERENCES nodes(id)
);
```

Indexes:

```sql
CREATE INDEX IF NOT EXISTS idx_skill_usage_feedback_skill ON skill_usage_feedback(skill_id);
CREATE INDEX IF NOT EXISTS idx_skill_usage_feedback_outcome ON skill_usage_feedback(outcome);
```

### 4.2 Outcome Enum

Canonical outcomes:

```text
success
partial
miss
misleading
trigger_mismatch
```

Backward-compatible rating mapping:

```text
helpful -> success
partially_useful -> partial
not_helpful -> miss
misleading -> misleading
```

No old API break is needed. `rating` can remain accepted, but implementation should convert it to `outcome` internally.

### 4.3 Artifact Fields

Prefer not to add many new columns. Use existing fields first:

- `trial_count`
- `trial_success_count`
- `trial_failure_count`
- `last_trial_at`
- `promotion_candidate`
- `needs_revision`
- `metadata`

Store audit metadata under `skill_artifacts.metadata`:

```json
{
  "usage_loop": {
    "last_audit_at": "...",
    "audit_failures": 0,
    "trigger_mismatch_count": 0,
    "last_failure_prompt_id": "..."
  }
}
```

Use columns only if query performance later requires it. v7.2 should stay small.

## 5. Store API Changes

### 5.1 Extend `skill_feedback`

Current signature:

```python
skill_feedback(skill_id, rating, note="", task_context="", used_as="trial", verification_result="")
```

Target signature:

```python
skill_feedback(
    skill_id: str,
    rating: str = None,
    note: str = "",
    task_context: str = "",
    used_as: str = "trial",
    verification_result: str = "",
    outcome: str = None,
    create_test_prompt: bool = False,
    expected: str = "",
    prompt_tags: list[str] = None,
) -> dict
```

Rules:

- Accept either `outcome` or legacy `rating`.
- If both are provided, `outcome` wins and `rating` is preserved for compatibility.
- Validate `used_as in {'approved', 'trial', 'experimental'}`.
- Write `skill_usage_feedback` row.
- Still write a `skill_feedback` node for graph search.
- Add the correct edge.
- Update counters and metadata.

### 5.2 Edge Mapping

```text
success -> verified_by
partial -> needs_revision
miss -> fails_on
misleading -> fails_on
trigger_mismatch -> needs_revision
```

Weights:

```text
success: 0.7
partial: 0.7
miss: 0.7
misleading: 0.9
trigger_mismatch: 0.8
```

### 5.3 Counter Rules

```text
Every feedback -> trial_count += 1 when used_as='trial'
success -> trial_success_count += 1
miss/misleading/trigger_mismatch -> trial_failure_count += 1
partial -> no success, no failure, but needs_revision=1
```

For approved skills, do not increment trial counters unless `used_as='trial'`. Approved usage should update structured feedback and audit metadata.

### 5.4 Prompt Creation

When `create_test_prompt=True`, call `add_skill_test_prompt(...)` and then `sync_skill_test_prompts_file(...)`.

Prompt ID format:

```text
usage-<short-feedback-id>
```

Prompt body:

```text
task_context
```

Expected field:

```text
expected if provided else verification_result
```

Tags:

```text
usage_feedback
outcome:<outcome>
regression    # for miss/misleading/trigger_mismatch
```

Safety rule:

```text
Do not auto-create a prompt when task_context is empty.
```

### 5.5 Audit Helpers

Add small helper methods, not a full scheduler:

```python
should_audit_skill(skill_id: str, trigger: str = "sampling") -> dict
record_skill_audit(skill_id: str, eval_result: dict, trigger: str = "sampling") -> dict
```

`should_audit_skill` is deterministic and policy-only. It should not call models.

Inputs:

```text
status
risk_level
latest_live_test_delta
latest_decision
trial_failure_count
metadata.usage_loop
trigger
```

Output:

```json
{
  "audit_required": true,
  "reason": "failure_triggered",
  "priority": "high"
}
```

`record_skill_audit` applies lifecycle changes after an evaluation:

```text
pass -> keep current status
fail on approved -> needs_revision
misleading/unsafe -> deprecated
```

No automatic mutation here. Audit records evidence; evaluation/mutation stays in v7.1 runner.

## 8. Tests

Add:

```text
tests/test_skill_evidence_flow.py
```

Use standard-library `unittest`, same style as `tests/test_bilateral_skill_evolution.py`.

Required tests:

1. `success` feedback creates `verified_by` and structured feedback row.
2. Legacy `rating='helpful'` maps to `outcome='success'`.
3. `partial` creates `needs_revision` without failure counter.
4. `miss` creates `fails_on`, increments failure counter, can create test prompt.
5. `misleading` creates high-weight `fails_on`, disables trial if used as trial, can create regression prompt.
6. `trigger_mismatch` records trigger mismatch metadata and creates `needs_revision`.
7. Approved usage does not increment trial counters.
8. Trial usage requires feedback before promotion candidate can matter.
9. `should_audit_skill(..., trigger='failure')` returns high-priority audit for approved skills.
10. `record_skill_audit` downgrades approved skill on failed audit.
11. v7.1 bilateral tests still pass.

Test isolation:

- Use temp DB.
- Redirect `_SKILLS_DIR` to temp directory.
- No network.
- No real API keys.

### Background Deployment

The smallest background deployment uses the existing APScheduler-based task runner and a dedicated daemon script:

```text
scripts/skill_daemon.py
skill-daemon.cmd
```

Defaults:

```text
full dream cycle: 03:00, 12:00, 17:00
skill audit scan: every 2 hours
```

Modes:

```text
--once  -> run one dream + audit pass and exit
--loop  -> keep the daemon alive after scheduling jobs
```

The daemon does not introduce a new service stack. It only schedules existing command-line entry points and audit scans.

Also verify no secrets:

```powershell
git status --short
git diff --stat
```

## 12. Acceptance Definition

v7.2 implementation is acceptable when this is true:

```text
Skill usage can produce structured evidence.
That evidence updates graph edges, counters, prompt files, and audit state.
Failures can become regression prompts.
Approved skills can be audited and downgraded.
No skill improves or stays approved merely because it was injected.
```
