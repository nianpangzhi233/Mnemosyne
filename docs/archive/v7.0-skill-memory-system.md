# Mnemosyne v7.0 Skill Memory System Blueprint

> Version: 7.0.0 | Date: 2026-05-09 | Status: Construction Blueprint
> Prerequisite: v6.1.0 released and verified
> Core idea: memories should not only be retrieved; mature experience clusters should crystallize into reusable, verified skills.

---

## 1. Goal

Mnemosyne v7.0 upgrades the validated Skill Loop demo into a formal Skill Memory System.

The goal is:

```text
experience cluster -> skill embryo -> LLM draft -> Darwin evolved skill -> approved skill -> safe injection -> feedback -> graph update
```

In one sentence:

```text
Mnemosyne v7.0 lets memory crystallize into skills, reuse them during tasks, verify the result, and write feedback back into the graph.
```

v7.0 is not an autonomous execution agent. It is a memory-to-skill governance system.

---

## 2. Scope

### In Scope

- Skill creation from memory nodes and experience clusters.
- Skill artifact storage and lifecycle management.
- Skill search and safe injection.
- Verification-gated approval.
- Trial injection for low-risk evolved skills.
- Feedback-driven skill promotion, revision, or deprecation.
- SKILL.md file mirrors for evolved and approved skills.
- Darwin evolution records and scoring integration.

### Out of Scope

- Browser automation.
- Desktop, ADB, WeChat, QQ, or IM automation.
- Generic script execution.
- Full autonomous agent runtime.
- Automatic external system modification.
- Visual skill editor.

Low-risk validation actions may exist internally, but v7.0 does not expose a generic executor.

## 3. Decisions

> Historical note: v7.0 used conservative manual approval language. Current v7.2+ design follows the feedback-first principle: middle layers hard-filter garbage and risk, while grounded generated artifacts enter executable evaluation loops directly. Human approval is reserved for default injection of risky or irreversible behavior, not for routine test-prompt generation.

| ID | Decision |
|----|----------|
| D1 | v7.0 only builds the Skill Memory System, not a generic execution agent. |
| D2 | Skill Crystallization is semi-automatic. Draft skills are not injected by default. |
| D3 | Add a `skill_artifacts` table. Skills become first-class artifacts. |
| D4 | `nodes` remains the primary record. `skill_artifacts` is a 1:1 structured extension. |
| D5 | Use the full lifecycle: `embryo -> draft -> evolved -> approved -> deprecated`. |
| D6 | `approved` defaults to manual approval. Strict auto-approval is reserved and disabled by default. Experimental auto-approval may be used for rule exploration, but must not enter normal injection by default. |
| D7 | Darwin evolution uses a dual rubric: Mnemosyne score plus Darwin score. |
| D8 | DB is the source of truth. `evolved` and `approved` skills must have a `SKILL.md` mirror. |
| D9 | Build the complete three-stage architecture, but validate it by milestones. |
| D10 | Default injection only returns approved skills. Low-risk evolved skills may be trial-injected with required feedback. |
| D11 | Trial feedback uses a hybrid collection model: first use and task-end prompts, with no repeated prompt for the same skill in one session. |
| D12 | Add `memory_skill_feedback` as a first-class tool. Do not fake feedback with `memory_write`. |
| D13 | Create complete foundation tables: `skill_artifacts` and `skill_evolution_runs`. |
| D14 | Skill file mirrors live at `skills/<slug>/SKILL.md`. |
| D15 | A skill must have at least one `verified_by` edge before it can be approved. |
| D16 | `deprecated` is soft deprecation only. Keep node, artifact, file mirror, and evidence chain. Hard purge is a separate safety process. |
| D17 | v7.0 requires five E2E acceptance tests. |
| D18 | v7.0 exposes six Skill MCP tools: crystallize, search, inject, approve, feedback, deprecate. |

## 4. Architecture

### 4.1 Three-Stage Skill Emergence

```text
Graph Embryo -> LLM Development -> Darwin Evolution -> Approval -> Injection -> Feedback
```

| Stage | Owner | Output | Default Injection |
|-------|-------|--------|-------------------|
| Graph Embryo | Graph clustering and Dream pipeline | `embryo` | No |
| LLM Development | LLM draft writer | `draft` | No |
| Darwin Evolution | Darwin optimizer and evaluator | `evolved` | No, except low-risk trial mode |
| Approval | Human or strict rule gate | `approved` | Yes, if `inject_enabled=true` |
| Deprecation | Feedback or manual decision | `deprecated` | No |

### 4.2 Role Separation

- Graph discovers patterns. It does not write polished skills.
- LLM develops drafts. It does not approve skills.
- Darwin evolves and scores. It does not grant production injection by itself.
- Approval gates injection. It must enforce `verified_by`.
- Feedback updates the graph and counters. It decides promotion candidates, revision needs, or deprecation pressure.

This separation prevents the GenericAgent-style failure mode where an LLM writes a good-looking SOP and the system treats it as a reliable skill.

## 5. Lifecycle

```text
embryo -> draft -> evolved -> approved -> deprecated
```

### embryo

A graph-discovered skill embryo.

Properties:

- Has source memory nodes.
- Has cluster reason.
- May have candidate triggers.
- Does not need a complete procedure.
- Does not inject.

### draft

An LLM-developed skill draft.

Properties:

- Has trigger patterns, preconditions, procedure, verification, and failure modes.
- Can be found by `memory_skill_search`.
- Does not inject by default.

### evolved

A Darwin-optimized skill.

Properties:

- Has Darwin evaluation records.
- Has scores.
- Must have `SKILL.md` mirror.
- Can be trial-injected only when low risk and `trial_enabled=true`.

### approved

A production skill allowed for default injection.

Requirements:

- `status='approved'`
- `inject_enabled=1`
- At least one `verified_by` edge.
- `SKILL.md` mirror exists.

### deprecated

A soft-deprecated skill.

Rules:

- Not injected.
- Not trialed.
- Node is kept.
- Artifact is kept.
- SKILL.md is kept and marked deprecated.
- Evidence edges are kept.

Hard deletion is not deprecation. It is a separate purge process for secret, privacy, or illegal-sensitive content.

## 6. Data Model

### 6.1 skill_artifacts

```sql
CREATE TABLE IF NOT EXISTS skill_artifacts (
    node_id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    slug TEXT UNIQUE,
    status TEXT NOT NULL DEFAULT 'draft',
    version TEXT DEFAULT '0.1.0',

    trigger_patterns TEXT DEFAULT '[]',
    preconditions TEXT DEFAULT '[]',
    procedure TEXT DEFAULT '[]',
    verification TEXT,
    failure_modes TEXT DEFAULT '[]',

    risk_level TEXT DEFAULT 'medium',
    review_status TEXT DEFAULT 'draft',
    approval_mode TEXT,
    inject_enabled INTEGER DEFAULT 0,
    trial_enabled INTEGER DEFAULT 0,
    requires_feedback INTEGER DEFAULT 0,

    mnemosyne_score REAL,
    darwin_score REAL,
    final_score REAL,

    source_node_ids TEXT DEFAULT '[]',
    evidence_node_ids TEXT DEFAULT '[]',

    trial_count INTEGER DEFAULT 0,
    trial_success_count INTEGER DEFAULT 0,
    trial_failure_count INTEGER DEFAULT 0,
    last_trial_at TEXT,
    promotion_candidate INTEGER DEFAULT 0,
    needs_revision INTEGER DEFAULT 0,

    file_path TEXT,
    file_hash TEXT,
    file_synced_at TEXT,

    created_at TEXT NOT NULL,
    updated_at TEXT,
    approved_at TEXT,
    deprecated_at TEXT,

    metadata TEXT DEFAULT '{}',

    FOREIGN KEY(node_id) REFERENCES nodes(id)
);
```

### 6.2 skill_evolution_runs

```sql
CREATE TABLE IF NOT EXISTS skill_evolution_runs (
    id TEXT PRIMARY KEY,
    skill_node_id TEXT NOT NULL,
    old_score REAL,
    new_score REAL,
    mnemosyne_score REAL,
    darwin_score REAL,
    status TEXT,
    dimension TEXT,
    note TEXT,
    eval_mode TEXT,
    created_at TEXT NOT NULL,
    metadata TEXT DEFAULT '{}',
    FOREIGN KEY(skill_node_id) REFERENCES nodes(id)
);
```

### 6.3 Indexes

```sql
CREATE INDEX IF NOT EXISTS idx_skill_status ON skill_artifacts(status);
CREATE INDEX IF NOT EXISTS idx_skill_inject ON skill_artifacts(inject_enabled);
CREATE INDEX IF NOT EXISTS idx_skill_trial ON skill_artifacts(trial_enabled);
CREATE INDEX IF NOT EXISTS idx_skill_slug ON skill_artifacts(slug);
CREATE INDEX IF NOT EXISTS idx_skill_promotion ON skill_artifacts(promotion_candidate);
CREATE INDEX IF NOT EXISTS idx_skill_evolution_skill ON skill_evolution_runs(skill_node_id);
```

### 6.4 Graph Edges

Required or reserved edge types:

| Relation | Meaning |
|----------|---------|
| `crystallized_from` | Skill came from source memory nodes. |
| `developed_from` | Draft came from embryo. |
| `verified_by` | Skill was validated by full test, explicit verification evidence, Darwin prompt, or helpful feedback. |
| `fails_on` | Skill failed or misled on a task. |
| `needs_revision` | Feedback indicates partial usefulness or required revision. |
| `supersedes` | New skill replaces old skill. |
| `deprecated_by` | Optional correction or decision node explains deprecation. |

## 7. File Mirrors

### 7.1 Path

```text
skills/<slug>/SKILL.md
```

Example:

```text
skills/windows-sqlite-encoding-check/SKILL.md
```

Optional companion files:

```text
skills/<slug>/metadata.json
skills/<slug>/test-prompts.json
skills/<slug>/evolution-card.png
```

### 7.2 Source of Truth

The database remains the source of truth.

`SKILL.md` is an artifact mirror used for:

- Inspection and debugging.
- Git diff.
- Darwin optimization.
- Export and sharing.

`skill_artifacts.file_hash` must match the current SKILL.md content after sync.

### 7.3 Deprecation Marker

Deprecated SKILL.md files must be kept and marked near the top:

```markdown
> Status: deprecated
> Deprecated reason: replaced by <slug or node id>
> Superseded by: node:<id>
```

## 8. Scoring

### 8.1 Final Score

```text
final_score = 0.6 * mnemosyne_score + 0.4 * darwin_score
```

### 8.2 Mnemosyne Rubric

Mnemosyne score evaluates whether a skill deserves to become long-term memory capability.

Suggested dimensions:

| Dimension | Weight |
|-----------|--------|
| Evidence chain completeness | 20 |
| Trigger precision | 15 |
| Procedure executability | 15 |
| Verification feasibility | 15 |
| Failure modes and boundaries | 10 |
| Injection safety | 10 |
| Graph governance and maintainability | 10 |
| Expression quality | 5 |

### 8.3 Darwin Rubric

Darwin score evaluates SKILL.md quality and measured task improvement.

It covers:

- Frontmatter and structure.
- Workflow clarity.
- Concrete checkpoints.
- Boundary conditions.
- Specific instructions.
- Resource integration.
- Overall architecture.
- Test prompt performance.

## 9. MCP Tool Surface

v7.0 exposes six Skill-specific MCP tools.

### 9.1 memory_crystallize

Purpose:

```text
memory nodes -> skill draft or embryo
```

Expected actions:

- Create `nodes.type='skill'`.
- Create `skill_artifacts` record.
- Add `crystallized_from` edges.
- Set `status='draft'` unless explicitly creating an embryo.
- Set `inject_enabled=0`.

### 9.2 memory_skill_search

Purpose:

Search skills across statuses. This is discovery, not safe injection.

Rules:

- Can filter by status.
- Can return draft, evolved, approved, and deprecated if requested.
- Must include status in results.

### 9.3 memory_skill_inject

Purpose:

Safe skill injection for the current task.

Default rules:

```text
status='approved' AND inject_enabled=1
```

Experimental rules:

- `mode='experimental'` may include draft and evolved skills.
- `mode='trial'` may include low-risk evolved skills when `trial_enabled=1`.
- Trial output must mark `feedback_required=true`.

### 9.4 memory_skill_approve

Purpose:

Approve a skill for default injection.

Hard requirements:

- Skill exists.
- Artifact exists.
- At least one `verified_by` edge exists.
- `SKILL.md` exists for evolved or approved skill.

Actions:

- Set `status='approved'`.
- Set `approval_mode`.
- Set `inject_enabled=1` unless approval mode is experimental.
- Set `approved_at`.

### 9.5 memory_skill_feedback

Purpose:

Record whether a skill helped during a real task.

Required inputs:

- `skill_id`
- `rating`: `helpful | not_helpful | misleading | partially_useful`

Optional inputs:

- `note`
- `task_context`
- `used_as`: `approved | trial | experimental`
- `verification_result`

Actions:

- Create `type='skill_feedback'` node.
- `helpful` creates `verified_by` edge.
- `not_helpful` or `misleading` creates `fails_on` edge.
- `partially_useful` creates `needs_revision` edge.
- Update trial counters and flags.

### 9.6 memory_skill_deprecate

Purpose:

Soft-deprecate a skill.

Actions:

- Set `status='deprecated'`.
- Set `inject_enabled=0`.
- Set `trial_enabled=0`.
- Set `deprecated_at`.
- Preserve node, artifact, SKILL.md, and evidence edges.

### 9.7 Not Exposed in v7.0

Do not expose these as normal MCP tools in v7.0:

- `memory_skill_purge`
- `memory_skill_auto_approve`
- `memory_skill_develop`
- `memory_skill_evolve`

Reasons:

- Purge is dangerous and must use a separate safety process.
- Auto-approval is reserved and disabled by default.
- Development and evolution can run as internal pipeline or scripts until stable.

## 10. Injection Policy

### 10.1 Default

```text
approved + inject_enabled=true
```

Do not inject:

- embryo
- draft
- evolved, unless trial mode allows it
- deprecated
- skills without `verified_by`

### 10.2 Trial Injection

Trial injection is allowed only when:

```text
status='evolved'
risk_level='low'
trial_enabled=1
requires_feedback=1
```

Trial injection output must include:

```text
status=evolved
used_as=trial
feedback_required=true
```

### 10.3 Trial Promotion

Suggested promotion candidate rule:

```text
trial_success_count >= 3
trial_failure_count = 0
verified_count >= 2
darwin_score >= 85
mnemosyne_score >= 85
```

Promotion candidate does not equal approved. D6 still applies.

### 10.4 Trial Failure

If rating is `misleading`:

- Set `needs_revision=1`.
- Set `trial_enabled=0`.
- Add `fails_on` edge.

If repeated misleading feedback occurs, deprecation may be recommended.

## 11. Milestones

### M1: Schema + Manual Crystallize + Search/Inject

Deliverables:

- Migration for `skill_artifacts` and `skill_evolution_runs`.
- `SQLiteStore` CRUD for skill artifacts.
- `memory_crystallize`.
- `memory_skill_search` backed by artifacts.
- `memory_skill_inject` with approved-only default gate.

Acceptance:

- Manual crystallization creates node, artifact, and edges.
- Draft skills are searchable but not injected.
- Approved skills are injected.

### M2: SkillEmbryoPhase

Deliverables:

- Graph clustering candidate discovery.
- Embryo creation from stable experience clusters.
- `crystallized_from` edges to source memories.

Acceptance:

- Candidate clusters generate embryos only.
- Embryos are not injected.

### M3: LLM Development

Deliverables:

- Embryo to draft conversion.
- Skill fields extracted: trigger, preconditions, procedure, verification, failure modes.
- Quality gate rejects non-executable or non-verifiable drafts.

Acceptance:

- LLM cannot approve skills.
- Drafts must include executable procedure and verification.

### M4: SKILL.md Mirror + Darwin Evolution

Deliverables:

- `skills/<slug>/SKILL.md` generation.
- File hash sync.
- Darwin score integration.
- `skill_evolution_runs` records.

Acceptance:

- Evolved skills have SKILL.md mirrors.
- Darwin runs update scores and evolution records.

### M5: Approval Gate + Feedback + E2E Tests

Deliverables:

- `memory_skill_approve`.
- `memory_skill_feedback`.
- `memory_skill_deprecate`.
- Five E2E acceptance tests.

Acceptance:

- Approval requires `verified_by`.
- Feedback updates graph edges and counters.
- Deprecated skills remain preserved but do not inject.

## 12. Acceptance Tests

### E2E 1: Manual Crystallization

Input:

```text
node_ids = [df79f577]
name = Windows SQLite Encoding Check
```

Expected:

```text
nodes.type=skill created
skill_artifacts created
crystallized_from edge created
status=draft
inject_enabled=false
```

### E2E 2: Verification-Gated Approval

Input:

```text
approve skill without verified_by
```

Expected:

```text
approve fails
```

Input:

```text
add verified_by edge
call memory_skill_approve
```

Expected:

```text
status=approved
inject_enabled=true
approved_at set
```

### E2E 3: Default Injection Only Returns Approved Skills

Setup:

```text
one draft skill
one evolved skill
one approved skill
one deprecated skill
```

Default injection expected:

```text
only approved + inject_enabled=true is returned
```

Experimental injection expected:

```text
draft/evolved may be returned only when explicitly requested
all returned skills include status
```

### E2E 4: Low-Risk Evolved Trial Injection and Feedback

Setup:

```text
status=evolved
risk_level=low
trial_enabled=true
requires_feedback=true
```

Trial injection expected:

```text
skill returned as trial
feedback_required=true
```

Helpful feedback expected:

```text
skill_feedback node created
verified_by edge created
trial_count += 1
trial_success_count += 1
last_trial_at updated
```

Misleading feedback expected:

```text
skill_feedback node created
fails_on edge created
trial_failure_count += 1
needs_revision=true
trial_enabled=false
```

### E2E 5: File Mirror and Soft Deprecation

Setup:

```text
skill reaches evolved or approved
```

Expected:

```text
skills/<slug>/SKILL.md exists
skill_artifacts.file_path set
skill_artifacts.file_hash set
file hash matches DB hash
```

After deprecation expected:

```text
status=deprecated
inject_enabled=false
trial_enabled=false
node preserved
artifact preserved
SKILL.md preserved and marked deprecated
evidence edges preserved
```

### Regression Test: FTS5 Special Character Fallback

Search input:

```text
PowerShell SQLite mojibake unicode_escape?
```

Expected:

```text
search does not crash on FTS5 syntax characters
falls back to vector-only or sanitized query when needed
```

## 14. Implementation Notes

- Keep edits minimal and compatible with v6.1.
- Do not remove existing `nodes.metadata` skill demo fields until migration is verified.
- `memory_skill_search` should move from raw `nodes.type='skill'` metadata to `skill_artifacts` when M1 lands.
- `memory_skill_inject` must enforce artifact status and injection gates.
- Approval must not be implementable via plain metadata update.
- Feedback must be a first-class path because it updates counters and edges.
- Query sanitization or fallback is required for FTS5 special characters.
- Windows Python execution should avoid fragile `python -c` quoting for complex migration scripts.

---

## 15. Non-Goals and Safety

v7.0 must not become a generic executor.

Do not add:

- Arbitrary shell execution from Skill tools.
- Browser or desktop control.
- External messaging automation.
- Automatic hard deletion.
- Automatic default injection for unapproved skills.

Secrets and privacy content must never be promoted into skills. If discovered inside a skill, use a separate purge workflow, not deprecation.

---

## 16. Summary

Mnemosyne v7.0 makes Skill a first-class memory artifact.

The core rule is:

```text
Only verified, approved skills can affect default task context.
```

The system can still explore:

```text
embryo -> draft -> evolved -> trial
```

But production reuse requires:

```text
verified_by + approved + inject_enabled
```

This keeps the system alive without letting it become a skill garbage dump.
