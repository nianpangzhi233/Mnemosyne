# Mnemosyne v6.1 Optimization Blueprint

> Version: 6.1.0 | Date: 2026-05-08 | Status: Construction Blueprint
> Prerequisite: v6.0.0 (all 48 acceptance criteria passed)
> Philosophy: Biomimetic three-layer architecture, never degrade design standards

---

## 1. Overview

v6.1 is a performance + quality upgrade on v6.0. No new features. Every optimization is rooted in biomimetic principles.

### Architecture Evolution

```
v6.0: Flat processing (one level does everything)
v6.1: Three-layer biosimilar architecture

Hippocampus Layer (write-time, fast)
  ├── Predictive validation with VectorIndex fast routing
  ├── Auto-association (weak similar_to edges on write)
  └── Reuse encoded vectors (zero redundant embed)

REM Sleep Layer (dream-time, medium)
  ├── Incremental SimilarTo (new nodes vs all)
  ├── Incremental Contradicts (diff-based, keyword + LLM)
  └── Lazy index rebuild (synaptic pruning)

Prefrontal Layer (LLM-augmented, slow but deep)
  ├── LLM contradiction judge (semantic understanding)
  ├── LLM predictive validation (scope-aware)
  └── Reuse existing llm_judge.py infrastructure
```

### Design Principles

| Principle | Human Brain Analogy | Code Impact |
|-----------|-------------------|-------------|
| Process new, reuse cached | Hippocampus only compares new input to indexed memories | Incremental diff for all batch phases |
| Fast route before deep route | Pattern completion before prefrontal deliberation | VectorIndex top-1 before full scan |
| Don't encode what's already encoded | Neurons don't re-fire for stored patterns | Reuse `vector` field, avoid redundant `embedder.encode()` |
| Prune, don't delete | Synaptic weakening before structural removal | Lazy index rebuild with stale ratio trigger |
| Creativity = association distance | Prefrontal cortex connects distant concepts | Creative mode seed expansion + is_a zero-hop |

## 2. Optimization 1: Three-Tier Contradiction Detection

### Problem

`ContradictsPhase` in v6.0 Dream Slow Path uses O(n²) pairwise comparison with 2 `embedder.encode()` calls per pair. On 215 nodes with Harrier CPU inference (cold: 9.5s, hot: 0.14s), this phase times out at >10 minutes.

The root cause is **full re-analysis of already-compared pairs** every Dream cycle.

### Biomimetic Rationale

The human brain does NOT scan all memories pairwise to find contradictions. Instead:

1. **Encoding-time**: Hippocampus flags conflicts immediately when new input mismatches stored predictions
2. **REM sleep**: Only recently activated memory traces are replayed, not the entire store
3. **Prefrontal deliberation**: Complex contradiction judgment uses semantic reasoning, not pattern matching

### Design

```
Hippocampus Layer (add_node, already exists in v6.0):
  precondition_index.search(top=1) → compare predicted_outcome → auto-contradict/verify
  → Works. Keep unchanged.

REM Layer (new ContradictsPhase):
  Step 1: Get nodes created since last_dream (diff)
  Step 2: For each new node, find top-10 similar existing nodes via VectorIndex
  Step 3: For each pair: keyword check (OPPOSITE_PAIRS, free)
  Step 4: For pairs passing keyword check: LLM judge (semantic, accurate)

Prefrontal Layer (new, optional):
  If LLM enabled: submit candidate pairs to llm_judge
  Returns: {contradicts: bool, scope: string, confidence: float}
  Reuses existing _call_llm / _extract_json infrastructure
```

### Code Changes

**File**: `scripts/core/dream_pipeline.py` — `ContradictsPhase.run()`

```python
def run(self, store, embedder) -> dict:
    last_dream = self._get_last_dream_time(store)
    new_nodes = store.query_nodes(
        "created_at > ? AND type = 'experience'",
        (last_dream,)
    )
    if not new_nodes:
        return {"added": 0, "evolved": 0, "scanned": 0}

    all_nodes = store.bulk_get_vectors()
    existing = store.bulk_get_edge_pairs("contradicts")

    # Build VectorIndex from all nodes for top-k
    store._ensure_vector_index()  # already built

    edges_to_add = []
    evolution_updates = []
    scanned = 0

    for new_n in new_nodes:
        new_vec = np.frombuffer(new_n.get("vector"), dtype=np.float32)
        candidates = store._vector_index.search(new_vec, top=10)
        for cand_id, sim in candidates:
            if (new_n["id"], cand_id) in existing or (cand_id, new_n["id"]) in existing:
                continue
            scanned += 1
            cand = store.get_node(cand_id)
            if not cand:
                continue

            # Step 1: Keyword fast-check (zero cost)
            if not self._has_opposition_keywords(new_n["content"], cand["content"]):
                continue

            # Step 2: LLM deep-check (if enabled)
            from llm_judge import load_config, _call_llm, _extract_json
            config = load_config()
            if config.get("enabled"):
                result = self._llm_judge_contradiction(
                    new_n, cand, config
                )
                if result and result.get("contradicts"):
                    edges_to_add.append({...})
                    evolution_updates.append({...})
            else:
                # No LLM: keyword match alone is sufficient signal
                edges_to_add.append({...})
                evolution_updates.append({...})

    added = store.bulk_add_edges(edges_to_add)
    for upd in evolution_updates:
        node = store.get_node(upd["id"])
        if node:
            new_conf = max(0.0, node.get("confidence", 1.0) + upd["delta"])
            store.update_node(upd["id"], confidence=new_conf)

    return {"added": added, "evolved": len(evolution_updates), "scanned": scanned}
```

**Complexity**: O(n_new × 10 × keyword_check) ≈ O(n_new) per Dream cycle. With ~10 new nodes/day, this is <1 second.

## 3. Optimization 2: VectorIndex Fast Routing for add_node Merge

### Problem

`add_node` line 118 performs a full-table scan of all experience node vectors in Python, doing pairwise dot products. At 200 nodes this is negligible (<1ms), but at 1000+ nodes it becomes a bottleneck. More importantly, it's **architecturally wasteful** — we already have a VectorIndex that does exactly this search.

### Biomimetic Rationale

The hippocampus doesn't scan every memory. It uses pattern completion — a fast, indexed lookup that finds the closest matching engram. The VectorIndex is our pattern completion circuit.

### Design

Replace the Python for-loop (lines 118-130) with a single VectorIndex lookup:

```python
# Before (O(n) Python loop):
cur.execute("SELECT id, vector FROM nodes WHERE vector IS NOT NULL AND type='experience'")
existing = cur.fetchall()
for nid, vec_blob in existing:
    ...

# After (O(1) indexed lookup):
self._ensure_vector_index()
candidates = self._vector_index.search(vector, top=1)
if candidates and candidates[0][1] > 0.92:
    best_id = candidates[0][0]
    # same merge logic...
```

The logic is identical — find the single most similar experience node, merge if >0.92. Only the implementation changes.

**File**: `scripts/core/sqlite_store.py` — `add_node()`, lines 113-150

Replace the `SELECT...fetchall()` + `for` loop block with `_ensure_vector_index()` + `_vector_index.search()`.

## 4. Optimization 3: Predictive Validation — Reuse Encoded Vectors

### Problem

In `add_node` Predictive Validation (lines 228-229), we call `self._embedder.encode()` twice — once for `old["predicted_outcome"]` and once for `content`. But `content` was **already encoded** on line 115 (`vector = self._embedder.encode(content)`). The second call is redundant.

### Design

```python
# Before:
old_pred_vec = self._embedder.encode(old["predicted_outcome"])
new_content_vec = self._embedder.encode(content)

# After:
old_pred_vec = self._embedder.encode(old["predicted_outcome"])  # still needed, not cached
new_content_vec = vector  # reuse already-encoded vector from merge step
```

Saves 1 `embedder.encode()` call per Predictive Validation cycle. At 0.14s hot inference, this is a small but free optimization.

### Code Changes

**File**: `scripts/core/sqlite_store.py` — `add_node()`, line 229

Change `self._embedder.encode(content)` to `vector` (the already-encoded variable from line 115 or 152).

## 5. Optimization 4: Creative Mode Semantic Jumps

### Problem

`search_spreading` precise and creative modes return identical results when the graph has few edges. The creative mode should emulate the brain's "free association" — connecting distant concepts through category hops.

### Biomimetic Rationale

When the brain thinks creatively, it doesn't just spread activation along direct connections — it jumps to **parent categories** (is_a edges) and spreads from there. This is why thinking about "torch installation" can trigger memories about "pip issues" — both are under the "Python dependency management" principle.

### Design

Creative mode enhancement:
```
Step 1: Standard seed + spread (keep existing)
Step 2: is_a zero-hop: for each seed node, find its is_a principle parent
Step 3: Include principle parent in activation map with initial activation = 0.5
Step 4: Continue spreading from principle parents (they have their own similar_to edges)
```

This creates a **two-hop semantic reach**: seed → principle → related experiences.

### Code Changes

**File**: `scripts/core/sqlite_store.py` — `search_spreading()`, creative mode branch

After seed initialization, add is_a parent injection:
```python
if mode == "creative":
    # Step 2.5: is_a zero-hop — inject principle parents
    for node_id in list(activation.keys()):
        cur.execute(
            "SELECT to_id FROM edges WHERE from_id=? AND relation_type='is_a' AND status='active'",
            (node_id,)
        )
        for row in cur.fetchall():
            principle_id = row[0]
            if principle_id not in activation:
                activation[principle_id] = 0.5  # moderate initial activation
```

## 7. Optimization 6: VectorIndex Lazy Rebuild (Synaptic Pruning)

### Problem

`delete_node` removes a node from SQLite but the VectorIndex still holds its vector and ID. Over time, deleted nodes accumulate as "ghost entries" in the index, wasting memory and producing incorrect search results (dead IDs).

### Biomimetic Rationale

The brain doesn't physically remove synapses immediately. It **weakens** them first (synaptic depression). Periodically during sleep, unused synapses are **pruned**.

### Design

```python
class VectorIndex:
    def __init__(self, dimension):
        ...
        self._deleted_ids = set()  # NEW

    def mark_deleted(self, node_id: str):
        self._deleted_ids.add(node_id)

    def search(self, query_vec, top=5):
        ...
        results = [(id, score) for id, score in raw_results if id not in self._deleted_ids]
        return results[:top]

    def rebuild_if_needed(self, nodes_fn):
        stale_ratio = len(self._deleted_ids) / max(1, len(self._id_map))
        if stale_ratio > 0.1:  # >10% ghost entries
            all_nodes = nodes_fn()
            self.build(
                vectors=np.array([n["vector"] for n in all_nodes]),
                ids=[n["id"] for n in all_nodes]
            )
            self._deleted_ids.clear()
```

`delete_node` calls `self._vector_index.mark_deleted(node_id)` (if index exists).

### Code Changes

**File**: `scripts/core/vector_index.py` — add `mark_deleted()`, `rebuild_if_needed()`, modify `search()` to filter
**File**: `scripts/core/sqlite_store.py` — `delete_node()` calls `mark_deleted()`; `_ensure_vector_index()` calls `rebuild_if_needed()` after build

## 8. Optimization 7: SimilarToPhase Incremental Detection

### Problem

`SimilarToPhase` in v6.0 Dream Fast Path is O(n²) pairwise comparison of ALL nodes with task_type. This is acceptable for 200 nodes (our benchmark showed <0.5s), but scales poorly. Applying the same "diff" principle as ContradictsPhase.

### Design

```python
class SimilarToPhase(DreamPhase):
    def run(self, store, embedder):
        last_dream = self._get_last_dream_time(store)
        new_nodes = store.query_nodes(
            "created_at > ? AND task_type IS NOT NULL AND type='experience'",
            (last_dream,)
        )
        if not new_nodes:
            return {"added": 0}

        all_nodes = store.bulk_get_vectors()
        all_nodes = [n for n in all_nodes if n.get("task_type") is not None]
        existing = store.bulk_get_edge_pairs("similar_to")

        new_ids = {n["id"] for n in new_nodes}

        edges_to_add = []
        # New vs All: only compare new nodes against the full set
        for new_n in new_nodes:
            va = np.frombuffer(new_n.get("vector"), dtype=np.float32)
            for old_n in all_nodes:
                if old_n["id"] == new_n["id"]:
                    continue
                if (new_n["id"], old_n["id"]) in existing:
                    continue
                vb = np.frombuffer(old_n.get("vector"), dtype=np.float32)
                sim = float(np.dot(va, vb))
                if sim > 0.85:
                    edges_to_add.append({...})

        added = store.bulk_add_edges(edges_to_add)
        return {"added": added}
```

**Complexity**: O(n_new × n_total) instead of O(n_total²). With ~10 new nodes/day and 200 total, this is ~2000 comparisons vs ~40000.

**File**: `scripts/core/dream_pipeline.py` — `SimilarToPhase.run()`

Add `_get_last_dream_time()` helper to `DreamPhase` base class (or `DreamPipeline`).

## 9. Optimization 8: Real-Time Auto-Association on Write

### Problem

Currently, `add_node` creates edges only in two cases:
1. principle match → is_a edge
2. precondition match → contradicts/verify

But the user gets **no immediate feedback** that this new memory relates to existing knowledge. The "encoding-retrieval coupling" that humans experience (writing a new memory instantly triggers related memories) is missing from the API response.

### Biomimetic Rationale

When the hippocampus encodes a new memory, it simultaneously activates related cortical representations. This is why new experiences instantly remind us of similar past ones — the encoding and retrieval circuits are coupled.

### Design

After `add_node` successfully writes a new node, add a **post-write association step**:

```python
# After INSERT, before return

# Find top-3 most similar existing nodes (fast, via VectorIndex)
self._ensure_vector_index()
related = self._vector_index.search(vector, top=3)

# Filter: only report if similarity > 0.7
strong_related = [(nid, sim) for nid, sim in related if sim > 0.7 and nid != node_id]

# Auto-create weak similar_to edges for strong matches
for related_id, sim in strong_related:
    self.add_edge(node_id, related_id, "similar_to",
                  weight=round(sim, 3), source="auto")

# Return related info in response
```

The MCP `memory_write` response can include a `related` field:
```json
{
  "node_id": "xxx",
  "merged": false,
  "related": [
    {"id": "yyy", "similarity": 0.85, "abstract": "torch 2.11 DLL crash..."},
    {"id": "zzz", "similarity": 0.73, "abstract": "pip install faiss timeout..."}
  ]
}
```

### Code Changes

**File**: `scripts/core/sqlite_store.py` — `add_node()`, add association step before return
**File**: `scripts/mcp_server/__init__.py` — include `related` in response

### Acceptance Criteria

|---|------|---------------|----------|
| 9.1 | add_node returns related memories | Write node with content similar to existing → response includes "related" array | P1 |
| 9.2 | Auto-created similar_to edges are weak | Check edge: source="auto", weight reflects similarity | P1 |
| 9.3 | No related returned when nothing similar | Write unique content → related array is empty | P1 |
| 9.4 | Merged node still returns related | Write node that merges → response includes related from the merged node's perspective | P2 |
