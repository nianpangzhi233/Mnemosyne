# Mnemosyne V8 Design Brief Draft

Status: draft for discussion.

This document summarizes the current V8 direction after the V7 archive decision and the first research pass. It is not an implementation plan yet.

## One-Sentence Direction

Mnemosyne V8 is an external, inspectable, governed memory kernel that turns raw experience into evidence-calibrated constraints over future generation and action.

## Non-Goals

V8 is not:

- A continuation of the V7 `nodes` / `edges` / `graph_dim` architecture.
- A GraphRAG system with more edge types.
- A generic knowledge graph.
- A complete agent runtime like Letta.
- A memory SaaS clone with `add()` / `search()` as the core abstraction.
- A skill-file warehouse.
- A parametric learning system that updates model weights.
- A biological-memory simulation.
- A system that treats LLM summaries as truth.

V8 must be honest about its substrate: it is non-parametric external memory. It can constrain and inform generation, but it does not make the base model internally learn unless a future consolidation path explicitly writes to adapters, finetunes, or model edits.

V8 should stay a memory kernel. It may integrate with agents, but it should not own agent identity, tool registration, orchestration, or full runtime state.

## Core Thesis

Memory is not the text that was retrieved.

For V8, memory is a governed state transition:

```text
Raw experience
-> candidate claim / procedure / preference / workflow
-> evidence calibration
-> scoped validated memory
-> read-gated ContextPack
-> observable runtime effect
-> revision / promotion / demotion / expiry
```

A summary generated by an LLM is only a candidate. It becomes usable memory only when it has source grounding, scope, lineage, evidence, and a current lifecycle status.

## Design Principles

- Raw trace is ground truth material, not memory by itself.
- LLM summary is a candidate claim, not validated memory.
- Every promoted memory needs source grounding and evidence.
- Lineage cannot be laundered by LLM rewriting.
- Read gates are as important as write gates.
- Procedural memory is control input and needs stricter governance than factual memory.
- Runtime trace is evidence; final answer alone is insufficient.
- Forgetting is governance, not cache cleanup.
- Scale must be measured with irrelevant-history growth, not assumed.
- External memory should not pretend to be parametric expertise.
- API names should not imply trust too early; `add_memory` is a dangerous abstraction if it bypasses evidence.

## Core Objects

### RawEvent

An immutable record of what actually happened.

Examples:

- User messages.
- Assistant messages.
- Tool calls.
- Tool outputs.
- File changes.
- Test results.
- Browser observations.
- Runtime errors.
- User corrections.

Required properties:

- Stable ID.
- Timestamp.
- Actor.
- Source type.
- Trust label.
- Session/task linkage.
- Raw content or pointer to raw content.
- Optional source span.

RawEvent must remain inspectable. Summaries may point to it but must not replace it.

### Manifest

A navigational aid over raw events.

Examples:

- Session summary.
- Trajectory index.
- Task timeline.
- Evidence map.
- File-change index.

Manifest is not truth. It helps locate evidence faster and must link back to RawEvent.

### Candidate

A proposed memory item derived from one or more RawEvents.

Candidate types:

- Claim: a factual or technical statement.
- Preference: a user or project preference.
- Constraint: a rule that should affect future behavior.
- Procedure: a how-to sequence.
- Workflow: a reusable multi-step routine.
- Skill: a higher-level executable procedural artifact.
- Gotcha: a known failure mode or trap.

Required properties:

- Type.
- Content.
- Source RawEvent IDs or spans.
- Lineage.
- Scope.
- Trigger / when-to-use.
- Preconditions.
- Risk level.
- Initial confidence.
- Status.

Candidate is not automatically injected into future context.

### Evidence

A record that supports, weakens, revises, or invalidates a Candidate or ValidatedMemory.

Evidence types:

- User confirmation.
- User correction.
- Task success.
- Task failure.
- Test result.
- Replay result.
- Validator result.
- Control-flow trace.
- Contradiction.
- Staleness signal.
- Poisoning/anomaly signal.

Evidence must be separate from the memory content. This keeps governance auditable.

### ValidatedMemory

A Candidate that has passed enough evidence and governance checks to be eligible for read-time use.

Required properties:

- Candidate content.
- Evidence summary.
- Evidence links.
- Scope.
- Status.
- Confidence.
- Freshness.
- Risk level.
- Read policy.
- Revision history.

ValidatedMemory can still be demoted, revised, expired, locked, or deleted.

### Procedure / Workflow / Skill

A high-risk subset of ValidatedMemory that may affect actions and tool use.

Required additional properties:

- Preconditions.
- Ordered steps.
- Allowed tools.
- Required validations.
- Forbidden shortcuts.
- Safe order constraints.
- Failure modes.
- Recovery steps.
- Control-flow evidence.

Procedural memory must not be promoted based only on a nice-looking document. It needs outcome evidence and preferably replay or reconstruction evidence.

### ContextPack

The read-time bundle injected into an LLM or agent.

ContextPack is produced by governance, not by raw vector top-k.

It may contain:

- ValidatedMemory items.
- RawEvent excerpts when fidelity matters.
- Evidence snippets.
- Procedure cards.
- Warnings.
- Scope limits.
- Freshness notes.

ContextPack should be typed, scoped, budget-aware, and safe to use for the current task.

### ContextSlot

A governed placement target for memory inside a prompt or agent context.

ContextSlot borrows the useful part of Letta-style memory blocks without adopting a full agent runtime.

Slot types:

- `always_visible`: pinned context, used sparingly for stable, high-value constraints.
- `retrieval_time`: included only when ReadGate selects it for the current task.
- `warning`: injected as a scoped caution, not as general knowledge.
- `procedure_card`: injected only when a workflow or skill is selected.

Required properties:

- Label.
- Scope.
- Max token budget.
- Allowed memory types.
- Freshness policy.
- Evidence requirement.
- Detach/expire policy.

Default should be `retrieval_time`, not `always_visible`. Always-visible slots are powerful but prone to staleness, token bloat, and prompt-level poisoning.

## Scope Model

V8 needs explicit scope on every event, candidate, memory, and evidence item.

Minimum scope fields:

- `user_id`: person or account the memory is about.
- `project_id`: project or workspace boundary.
- `agent_id`: agent or tool persona that produced or uses the memory.
- `session_id`: conversation or execution session.
- `task_id`: explicit task or job.
- `source_id`: file, URL, tool, log, or external system source.

Scope serves three jobs:

- Retrieval: find the right memory without pulling irrelevant projects or users.
- Governance: prevent cross-user, cross-project, and cross-agent leakage.
- Lifecycle: allow session-level expiry without deleting project-level knowledge.

Mem0's `user_id` / `agent_id` / `app_id` / `run_id` model is a useful engineering precedent, but V8 should use names that match this project: `project_id` instead of `app_id`, and `session_id` instead of `run_id`.

## Lifecycle

```text
observe -> encode -> propose -> validate -> promote -> retrieve -> apply -> observe outcome -> revise
```

States:

- `raw`: immutable source material only.
- `candidate`: proposed but not trusted.
- `validated`: eligible for read-time use within scope.
- `promoted`: high-value memory with strong evidence.
- `locked`: protected from automatic mutation.
- `stale`: freshness degraded.
- `demoted`: retained but normally not injected.
- `deprecated`: known to be superseded or risky.
- `deleted`: removed according to policy.

Operations:

- `encode`: convert raw input into a typed RawEvent or Candidate.
- `retrieve`: find candidates for a task.
- `promote`: raise lifecycle status based on evidence.
- `demote`: lower status after failures, contradiction, or staleness.
- `revise`: create a new version while preserving history.
- `merge`: combine related memories while preserving lineage.
- `split`: break over-broad memory into scoped units.
- `lock`: prevent automatic mutation.
- `expire`: enforce lifecycle timeout.
- `delete`: remove by policy.

## Write Governance

WriteGate decides what can enter Candidate or ValidatedMemory layers.

Checks:

- Is there a source RawEvent?
- Is the claim grounded in a source span?
- Is the scope explicit?
- Is the trigger specific enough?
- Is the memory too broad or overgeneralized?
- Does it contradict protected memory?
- Does it inherit untrusted lineage?
- Is it procedural or safety-sensitive?
- Is the evidence strong enough for the requested status?

WriteGate should be strict for procedures, workflows, skills, preferences, and safety rules.

LLM-powered inference must stop at Candidate creation. A background consolidator may propose memory updates, but it must not directly create ValidatedMemory without evidence and governance checks.

## Read Governance

ReadGate decides what may enter the current ContextPack.

Checks:

- Is the memory in scope for this project/user/task?
- Is it fresh enough?
- Is the lineage trusted enough for this use?
- Is there conflicting memory?
- Is raw evidence needed instead of a summary?
- Would injecting it create unsafe tool or workflow pressure?
- Does it fit the token and call budget?
- Does it require a warning or scope note?

ReadGate must be context-sensitive. Some memories are harmless in isolation but harmful when paired with a specific task.

ReadGate should build ContextPacks, not raw top-k search results. Search is only candidate retrieval; governance decides final inclusion and placement.

## API Shape

V8's API should expose the lifecycle instead of hiding it behind `add_memory`.

Core endpoints or command groups:

- `/events`: append immutable RawEvents.
- `/manifests`: build or inspect navigation aids over event ranges.
- `/candidates`: create or list candidate claims, preferences, procedures, workflows, and gotchas.
- `/evidence`: attach evidence to candidates or memories.
- `/memories`: inspect governed ValidatedMemory records.
- `/context-packs`: build read-gated ContextPacks for a task.
- `/lifecycle`: promote, demote, revise, split, merge, lock, expire, or delete.

Example write flow:

```text
POST /events
-> POST /candidates
-> POST /evidence
-> POST /lifecycle/promote
```

Example read flow:

```text
POST /context-packs/build
-> typed retrieval
-> ReadGate filtering
-> ContextSlot placement
-> ContextPack output
```

API design rule: raw input, inferred candidate, validated memory, and injected context must be separate objects.

## Procedural Safety

Procedural memory changes behavior. It can hijack control flow if poisoned or overtrusted.

V8 should treat the following as first-class deviation types:

- Insertion: an unauthorized step or tool is added.
- Omission: required validation or logging is skipped.
- Substitution: a safe tool or step is replaced.
- Reordering: valid steps run in an unsafe order.
- Branch injection: a hidden branch is introduced.
- Parameter manipulation: the tool is right but arguments are unsafe.
- Tool-result poisoning: a tool result corrupts later planning.

For high-risk workflows, V8 should represent a workflow envelope:

- Expected steps.
- Allowed tools.
- Required checks.
- Forbidden transitions.
- Parameter constraints.
- Evidence required before promotion.

## Evaluation Stack

V8 should be evaluated as a memory system, not a database.

### Task Effectiveness

- Task success rate.
- Factual correctness.
- Plan completion.
- Step count.
- Token cost.
- Latency.

### Memory Quality

- Retrieval precision/recall.
- Source-grounding coverage.
- Contradiction rate.
- Stale-memory injection rate.
- Evidence completeness.
- Read usefulness.

### Lifecycle Ability

- Test-time learning.
- Revision after failure.
- Selective forgetting.
- Cross-session coherence.
- Cold-start coverage.
- Usable-scale boundary.

### Procedural Safety

- Tool-choice deviation.
- Workflow-order deviation.
- Skipped validation.
- Unsafe shortcut rate.
- Poisoned memory activation.
- Suspicious side-effect rate.

## Research Anchors

This draft is informed by these research directions:

- Memp: procedural memory as editable repository built from trajectories.
- AWM: reusable workflow induction with parameterized subtask routines.
- ExpeL: success/failure experience distilled into rules of thumb.
- Faithfulness studies: agents often rely more on raw trajectories than condensed insights.
- MemoryAgentBench: accurate retrieval, test-time learning, long-range understanding, selective forgetting.
- Evo-Memory: search-predict-evolve protocol for procedural memory.
- MemoryArena: multi-session interdependent tasks.
- LongMemEval-V2 / AgentRunbook-C: evidence collection with manifests and trajectory inspection.
- MemLineage: lineage survives LLM-mediated derivation.
- A-MemGuard / MemSAD: isolated memory auditing is insufficient.
- MIND-Skill / SkillRouter / SkillsBench: skill generation needs verification and routing.
- MemOS / Text2Mem: memory as governed resource with lifecycle operations.
- ToolHijacker / ClawGuard / AgentDojo / SHADE-Arena: tool and workflow safety need trace-level evaluation.
- ROME / MEMIT / parametric-memory critique: external memory is not weight learning.
- Mem0: small memory APIs and entity-scoped filtering are useful, but inferred memory must not become truth automatically.
- Letta: core memory blocks versus archival memory is a useful separation, but V8 should not become a full agent runtime.
- LangMem: background consolidation and storage-agnostic memory tools are useful, but LLM memory managers must only propose candidates.

## MVP Cut

The first V8 prototype should avoid V7-scale complexity.

MVP should include:

- Immutable RawEvent log.
- Manifest layer for navigation.
- Candidate claim/procedure schema.
- Evidence ledger.
- Basic WriteGate.
- Basic ReadGate.
- ContextPack builder.
- Lifecycle status transitions.
- Scope model across user/project/agent/session/task/source.
- Minimal API groups for events, candidates, evidence, memories, context packs, and lifecycle operations.
- Minimal evaluation harness for retrieval, revision, forgetting, and procedural trace deviation.

MVP should not include:

- Full graph ontology.
- Autonomous dream pipeline.
- Parametric model editing.
- Multi-agent marketplace.
- Automatic skill approval.
- Complex RL router.
- Always-visible memory blocks by default.
- Agent-owned unrestricted memory rewriting.

## Open Questions

- What is the smallest useful RawEvent schema?
- Should RawEvent be file-backed, SQLite-backed, or both?
- How should source spans be represented for chat, files, tool outputs, and browser traces?
- What confidence model is simple enough for MVP but not fake precision?
- Which memory types are allowed into ContextPack by default?
- What should require explicit user confirmation?
- How should V8 measure whether a memory actually influenced the result?
- How much procedural safety should be deterministic versus LLM-judged?
- What is the first real benchmark task suite for V8?
- Which ContextSlots, if any, should be allowed to be always visible?
- What should the first public API call be named so it does not imply premature trust?

## Working Definition

Mnemosyne V8 is a memory kernel that preserves raw experience, derives scoped candidates, calibrates them with evidence, governs read/write access, and returns task-specific ContextPacks that are useful, auditable, and safe under scale.
